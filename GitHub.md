
## 🚀 初めてGitHubにPushするときの3ステップ
新しく作ったプロジェクトを、初めてGitHubにPushする場合は以下の手順になります。
### 1. GitHubでリポジトリを作る
GitHubにブラウザでログインし、「New repository」から空のリポジトリ（READMEや.gitignoreは追加しない設定）を作成します。作成すると、画面に以下のような**URL（アドレス）**が表示されます。
> https://github.com/あなたのユーザー名/リポジトリ名.git
> 
### 2. リモートリポジトリを登録する（初回のみ）
ターミナルでアプリのプロジェクトフォルダに移動し、上記のURLをGitに登録します。
```bash
git remote add origin https://github.com/あなたのユーザー名/リポジトリ名.git

```
 * ※ origin というのは、「アップロード先のデフォルトの場所」につける一般的なニックネームです。
### 3. いざ、Pushする！
準備ができたら、以下のコマンドで手元のコードを丸ごとGitHubに送信します。
```bash
git push -u origin main

```
 * ※ 最近のGitやXcodeのデフォルトのブランチ（親玉の枝）の名前は main です。もし古いプロジェクトなどでブランチ名が master になっている場合は、git push -u origin master になります。
 * ※ -u を最初につけておくと、次回からは git push とだけ打てば自動的に origin main に送られるようになります。
## 🔄 2回目以降のPush
一度上の設定が終われば、aiderがコードを書き換えてコミットした後にPushしたくなったら、ターミナルでこれだけを打てばOKです。
```bash
git push

```

わざわざaiderを終了したり、別のターミナルタブを開くのが面倒なときは、aiderのチャット画面で以下のように打ち込んでみてください。
```text
/run git push

```
これで、aiderが動いた状態のまま、その場でGitHubへのPushが完了します。便利なのでぜひ覚えておいてください！


エラーが出た

Git pushのエラー、お困りですね。画面を見る限り、実は**2つの異なる問題**が同時に発生しています。
 1. **メインのエラー：GitHub上の変更とローカルの競合（[rejected] main -> main (fetch first)）**
 2. **サブのエラー：Macの資格情報（キーチェーン）のエラー（failed to get/store: -25308）**
まずは、Gitの動作を正常にするためのメインのエラーの解決から進めましょう！
## 1. メインのエラーの解決法（競合の解消）
**原因：** GitHub（リモート）側に、あなたの手元のPC（ローカル）にはない新しいコミットが存在しているため、安全のためにプッシュが拒否されています。（GitHub上で直接ファイルを編集したり、他の端末からプッシュしたりしませんでしたか？）
### 解決手順
まずはリモートの最新状態を取り込み、手元の変更と合体（マージ）させます。ターミナルで以下のコマンドを順番に実行してください。
```bash
# 1. リモートの最新情報を取得
git fetch origin

# 2. リモートのmainブランチの変更を手元にマージ
git pull origin main

```
> 💡 **※もし git pull の時にエラーが出たら？**
> 「履歴に共通点がない（fatal: refusing to merge unrelated histories）」と言われた場合は、以下のコマンドを試してください。
> git pull origin main --allow-unrelated-histories
> 
git pull が成功し、ファイルに競合（コンフリクト）がなければ、そのままもう一度プッシュを試します。
```bash
git push origin main

```
## 2. サブのエラーの解決法（-25308 エラー）
プッシュの最初と最後に出ている failed to get: -25308 や failed to store というエラーは、**Macのキーチェーン（パスワード管理システム）がロックされているか、Gitがうまくアクセスできていない**ときに出るMac特有のエラーです。
これが原因で、毎回ユーザー名とパスワード（現在はパスワードではなく「パーソナルアクセストークン」が必要です）を聞かれている状態になっています。
### 解決手順
**① キーチェーンのロックを解除する**
Macをスリープから復帰させた直後などに頼まれることがあります。一度ターミナルで以下を実行してキーチェーンを明示的にアンロックしてみてください（Macのログインパスワードを求められます）。
```bash
security unlock-keychain

```
**② Gitの認証ヘルパーを再設定する**
Mac標準のキーチェーンを使うようにGitの設定を上書きします。
```bash
git config --global credential.helper osxkeychain

```
これで次回以降、一度正しく認証（ユーザー名とトークンを入力）すれば、-25308 エラーが出なくなり、パスワードの入力も省略できるようになります。
まずは **1の git pull origin main** から試してみてください。うまくいかない場合は、途中で出たエラーメッセージを教えていただければさらにサポートします！

なるほど、枝分かれ（divergent branches）が発生していて、Gitが「どうやって2つの履歴を合流させればいい？（マージにする？リベースにする？）」と迷ってストップしてしまった状態ですね。
一番シンプルで安全な**「マージ（Merge）」**という方法で合流させましょう。
以下の手順でコマンドを実行してください。
### 解決手順
**① まず、合流方法を「マージ」に指定して pull する**
以下のコマンドを実行して、強制的にマージの形でリモートの変更を取り込みます。
```bash
git pull origin main --no-rebase

```
**② 画面が変わった場合の対応**
上記のコマンドを打つと、お使いの環境によっては「マージコミットのメッセージ」を書くために画面（エディタ）が切り替わる場合があります。
 * **もし nano というエディタ（画面下に ^X Exit などと書いてある）が開いた場合：**
   そのまま Ctrl + O（保存）→ Enter → Ctrl + X（閉じる）で抜けられます。
 * **もし Vim というエディタ（画面左下に ~ が並んでいる）が開いた場合：**
   キーボードでそのまま :wq と入力して Enter を押すと保存して抜けられます。
**③ 最後にプッシュする**
無事に pull が完了したら、いつものようにプッシュすれば完了です！
```bash
git push origin main

```
### 💡 今後のためのワンポイント（任意）
毎回この警告が出るのを防ぎたい場合は、以下のコマンドを実行しておくと、「次から git pull するときは自動的にマージするよ」という設定をPCに保存できます。
```bash
git config --global pull.rebase false

```
まずは git pull origin main --no-rebase を試してみてください！

「**履歴に共通点がない（refusing to merge unrelated histories）**」というエラーですね。
これは、ローカル（手元）のGitリポジトリと、GitHub（リモート）のリポジトリが、それぞれ**全くの別物として新しく作られた（最初のコミットが異なる）**ことが原因です。Gitが「これ本当に同じプロジェクト？上書きしちゃって大丈夫？」と安全のためにブロックしてくれています。
今回は同じ OmikujiApp というプロジェクトで間違いないはずですので、**「関係ない履歴だけど、強制的に合体させてOK！」**という指示を出して解決します。
以下の手順で進めてみてください。
### 解決手順
**① 履歴の関連付けを強制して pull する**
以下のコマンドを実行して、異なる履歴の合流を許可します。
```bash
git pull origin main --no-rebase --allow-unrelated-histories

```
> 💡 **注意**：これを実行すると、マージコミットのメッセージを入力する画面（エディタ）が開く可能性が高いです。
>  * **nano（画面下に ^X Exit などがある）の場合**： Ctrl + O → Enter → Ctrl + X
>  * **Vim（左下に ~ が並んでいる）の場合**： :wq と入力して Enter
> 
**② 競合（コンフリクト）が発生しなかったか確認する**
もし無事に pull が終わればOKですが、もし同じファイル（例：README.md など）が双方にあって衝突した場合は、ターミナルに CONFLICT (content): Merge conflict in... と表示されます。その場合はファイルを修正する必要がありますが、まずは上記のコマンドを実行してみてください。
**③ 最後に push する**
無事に合流できたら、満を持してプッシュします！
```bash
git push origin main

```
まずは --allow-unrelated-histories を付けた pull を試してみてください！
